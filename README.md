# Git workflow standard for interns

## Термины:

- *Trunk-based development* — разработка вокруг основной ветки.
- *Branch protection* — защита ветки.
- *CI (Continuous Integration)* — непрерывная интеграция.
- *Push* — отправка изменений.
- *Pull Request* — «запрос на слияние изменений».
- *Code Review* — проверка кода.
- *Merge* — слияние веток.
- *History rewrite* — переписывание истории.
- *Squash merge* — слияние с объединением коммитов в один.
- *Atomic change* — неделимое логическое изменение.

## Архитектурные принципы:

- *Task-level detailed history* — детальная история в рабочей ветке задачи.
- *Action-level integration history* — агрегированная история по типам действий.
- *Atomic PR discipline* — атомарность: один файл = один рабочий коммит.
- *Deterministic commit grammar* — фиксированный формат подписей коммитов.

## Стратегия ветвления:

- Разработка вокруг основной ветки `main`
- Защита основной ветки `main`
- Короткоживущие `feature-ветки`
- Автоматические `CI` проверки
- Обязательный `Pull Request`
- Обязательный `Code Review`
- Обязательный `Squash Merge`

## Регламент:

>Настоящий регламент определяет правила работы с ветками в sandbox-репозитории (изолированная учебная среда).

- Стандартизация процесса разработки
- Запрет на изменения в основной ветке `main`
- Ветка `main` доступна только для чтения (read-only)
- Запрет на `push` в основную ветку `main`
- Запрет на `merge` в основную ветку `main`
- Изоляция решений интернов

## Основные ветки интернов:

>Постоянные именные персональные главные ветки, являющиеся аналогами ветки `main` и ответвлениями от ветки `main`.

- **Формула именования веток:**
    - `intern-<username>-<hash6>`
    - `intern-ivanov-xstqsb`
    - `intern-petrov-dvlewe`

- **Правила:**
    - Создаются один раз от `main`
    - Являются постоянными
    - Используются как персональные основные ветки
    - Являются персональными аналогами ветки `main`

```sh fold unwrap title:"Основные ветки интернов"
main (read-only)
├── intern-ivanov-xstqsb
└── intern-petrov-dvlewe
```

## Рабочие ветки интернов:

>Персональные рабочие ветки, создаваемые от персональной основной ветки интерна и предназначенные для выполнения интерном отдельных задач.

- **Формула именования веток:**
    - `intern-<username>-<hash6>/<task-number>-<task-type>-<task-name>`
    - `intern-ivanov-xstqsb/12-feature-admin-service`
    - `intern-petrov-dvlewe/13-bugfix-user-service`

- **Правила:**
    - Создаются от персональной основной ветки интерна
    - Предназначены для реализации одной задачи
    - После завершения сливаются в персональную основную ветку
    - Не сливаются в `main`
    - Не удаляются после слияния и не переиспользуются

```sh fold unwrap title:"Рабочие ветки интернов"
main (read-only)
├── intern-ivanov-xstqsb
│   ├── 12-feature-admin-service
│   ├── 13-refactor-admin-service
│   ├── 14-improvement-admin-service
│   └── 15-bugfix-admin-service
└── intern-petrov-dvlewe
    ├── 22-feature-user-service
    ├── 23-refactor-user-service
    ├── 24-improvement-user-service
    └── 25-bugfix-user-service
```

## Типы задач (task-type):

>`<task-type>` — это существительное, которое определяет тип задачи и отвечает на вопрос: «К какой категории относится данная задача?».

- **Типы:**
    - `feature` — новая функциональность
    - `refactor` — переработка без изменения логики
    - `improvement` — улучшение с изменением логики
    - `bugfix` — исправление ошибки

## Названия задач (task-name):

>`<task-name>` — это существительное, которое представляет объект, является кратким  описанием сути задачи и отвечает на вопрос: «Что именно создаётся или изменяется?».

- **Правила:**
    - Без повторения `task-type`
    - Только `lowercase` (нижний регистр)
    - Только латиница
    - Длина — не более 3–5 слов
    - Слова разделяются через `-`
    - Без предлогов и лишних слов
    - Без глаголов в прошедшем времени
    - Без служебных слов (`the`, `a`, `fix`, `update` и т.д.)

- **Примеры:**
    - `<object-name>`
    - user-controller (services/user/controller.js)
    - user-service (services/user/service.js)
    - user-model (services/user/model.js)
    - accounting-transformer (toAccounting.js)
    - age-checker (checkAge.js)

## Правила оформления коммитов:

>Настоящий раздел определяет правила организации истории изменений и стандарты подписи коммитов в Pull Request.

- **Принципы организации коммитов:**
    - Один изменённый файл = один рабочий коммит для PR
    - Один рабочий коммит отражает одно логическое изменение одного файла
    - Каждый рабочий коммит для PR должен быть атомарным
    - Запрещено добавлять 2 и более файлов в один рабочий коммит для PR
    - Разрешено добавлять 2 и более файлов в один интеграционный коммит

- **Цель:**
    - Прозрачная и навигационная история изменений
    - Удобство Code Review
    - Контролируемая декомпозиция изменений
    - Корректное отображение Last commit в GitHub

- **Двухуровневая модель истории:**
    - История состоит из двух уровней
    - Рабочий уровень — ветка задачи, детальная история изменений
    - Интеграционный уровень — основная ветка, агрегированная история по действиям

- **Правила работы в Pull Request:**
    - В процессе разработки допускается несколько рабочих коммитов
    - Каждый рабочий коммит подписывается по установленному формату
    - Коммит фиксирует тип действия, а не название модуля
    - После одобрения (approve) выполняется редактирование истории (rebase)
    - Коммиты группируются по выполненным действиям над конкретными файлами
    - Интеграционный коммит отражает один тип действия и может включать N файлов
    - Ветка задачи сохраняется для просмотра полной истории

- **Формат подписи рабочих коммитов:**
    - `<Глагол действие в прошедшем времени> <существительное>`
    - `<VerbPast> <EntityType>`
    - `VerbPast` — глагол в прошедшем времени
    - `EntityType` — тип сущности (module, config, test и т.д.)

- **Допустимые глаголы (действия):**
    - *Added* — добавлен
    - *Updated* — обновлён
    - *Refactored* — переработан
    - *Improved* — улучшен
    - *Fixed* — исправлен
    - *Renamed* — переименован
    - *Moved* — перемещён
    - *Deleted* — удалён

- **Примеры подписей рабочих коммитов:**
    - *Added module* (строго 1 добавленный модуль)
    - *Updated module* (строго 1 обновленный модуль)
    - *Refactored module* (строго 1 переработанный модуль)
    - *Improved module* (строго 1 улучшенный модуль)
    - *Fixed module* (строго 1 исправленный модуль)
    - *Renamed module* (строго 1 переименованный модуль)
    - *Moved module* (строго 1 перемещённый модуль)
    - *Deleted module* (строго 1 удалённый модуль)

- **Примеры подписей объединённых коммитов:**
    - *Added module* (от 1 до N добавленных модулей)
    - *Updated module* (от 1 до N обновленных модулей)
    - *Refactored module* (от 1 до N переработанных модулей)
    - *Improved module* (от 1 до N улучшенных модулей)
    - *Fixed module* (от 1 до N исправленных модулей)
    - *Renamed module* (от 1 до N переименованных модулей)
    - *Moved module* (от 1 до N перемещённых модулей)
    - *Deleted module* (от 1 до N удалённых модулей)
